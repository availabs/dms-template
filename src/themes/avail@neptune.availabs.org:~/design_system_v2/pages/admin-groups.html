Groups — permission sets that govern access.

Each group holds a named permission level (admin, editor, viewer, public) and a list of members. Sites filter their data and UI by checking a user's group membership.

SECTION · PERMISSION MATRIX

What each level can do.

Permission levels are set per group when the group is created. The level governs what authenticated members can see and do across all sites that check group membership.

DESIGN NOTES · GAPS VS. CURRENT

What this design adds over the current implementation.

Added
  • + Permission level badge (admin/editor/viewer/public) per group
  • + Avatar stack — member previews with initials and overflow count
  • + System vs. custom badge — system groups (Admin, public) are non-deletable
  • + "Show permissions" expander per row
  • + Permission matrix section — full capability comparison across all levels
  • + Disabled delete/edit for system groups
  • + Sidenav drill-down by permission level
Implementation notes
  • → Permission level: infer from group name ("Admin" → admin, "Editor"/"Steward" → editor, etc.) or add explicit auth_level field to group
  • → Avatar initials: first 2 chars of email or name, colour from index mod 4
  • → System groups: "Prod Admin" and "public" — treat groups whose name ends in " Admin" or equals "public" as system
  • → Member list: expand row triggers a GET to /groups/byproject filtered by group name, showing full member list
