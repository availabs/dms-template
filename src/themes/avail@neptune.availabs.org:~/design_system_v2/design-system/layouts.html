design system / layouts

Layouts — the page chrome.

Layout and LayoutGroup are the two primitives every DMS page is built from. They have no rich content of their own — they're the bands and surfaces every Section sits inside. The brand decides which variants exist; authors pick which one each page or band uses.

⌘ note → this page documents page chrome (Layout + LayoutGroup variants). For the page-content column grid that Sections lay on, see grid.html.
SECTION · HIERARCHY

The five layers.

Every DMS surface lives somewhere inside this nest. The brand styles the layers; authors arrange Sections inside them. Nothing in DMS sits outside this hierarchy.

Site
routing · ThemeContext · theme resolution
supplies the theme; styled only via theme overrides
↓
Layout
page chrome: outer wrapper, optional TopNav, optional SideNav, content area
3 wrapper class keys (outerWrapper, wrapper, wrapper2, wrapper3, childWrapper) + nav configuration
↓
LayoutGroup
a band within a Layout: header / content / footer / auth
3 wrapper class keys (wrapper1, wrapper2, wrapper3) per named variant
↓
Section
an editable block: title, group membership, the component instance it wraps
section chrome from the pages pattern theme
↓
Component
the actual content: Card, Spreadsheet, Graph, Map, Lexical, Header, …
each primitive's own theme keys (see components)
⌘ the contract → a brand styles the chrome and the primitives; authors compose pages by picking which LayoutGroup variants to use and which Sections to drop inside. There are no bespoke per-page layouts because there are no escape hatches from the hierarchy. The constraint is the feature.

SECTION · LAYOUTGROUP VARIANTS

LayoutGroup — four surface recipes.

A LayoutGroup is a single band of content. Each named variant is a different surface recipe: boxed vs. unboxed, hairline vs. flat, padding density, background. Authors pick which recipe for each band on a page.

SECTION · NESTING

Nesting — a real page stacks LayoutGroups.

A page is one Layout containing one or more LayoutGroups. Different bands use different variants — header for the title, content for the body, footer at the bottom. This is what every page in pages/ does.

SECTION · WHICH TO USE

Pick the variant; the chrome composes itself.

Reference for authors and designers: which Layout / LayoutGroup variant to use for which page kind.

Page kind Layout variant LayoutGroup variants typically stacked
Marketing homepage default header (hero) → content (×3–5) → footer
Long-form / Theory page default header (title) → content (single, narrow measure) → footer
Dashboard / analytics app content (header card) → content (chart cards stacked or in grid)
Dataset / table view app content (filter chrome) → content (table card)
Page editor app Editor chrome lives in the SideNav; content hosts the live preview
Sign-in / sign-up bare auth (single, centred)
Embedded / print bare content (single, no chrome)
