v0.2 design system · page chrome

Layouts — every page is a Layout, a stack of LayoutGroups, and the Sections inside them.

Layout owns the page chrome (outer wrapper, optional nav rails, the content area). LayoutGroups are the bands that stack inside — header, content, dark code, auth, footer. Sections live inside groups. The grammar is fixed; the brand decides only how each variant looks. Everything in components and patterns renders inside this structure.

See the Layout variants → Read the column grid →
scope → This page documents page chrome shapes — Layout + LayoutGroup variants, the bands that stack down a page. For the page-content column grid that Sections sit on (the 12-column sectionArray), see grid.html.
Hierarchy

Site → Layout → LayoutGroup → Section → Component.

Five layers. Every surface in DMS lives somewhere inside this nest — marketing homepage, dataset admin page, analytics dashboard, map viewer, sign-in screen. The same hierarchy renders all of them; the difference is which Sections appear in which LayoutGroups under which Layout. Below: the same diagram a brand reads on day one.

The hierarchy itself is fixed — an author cannot invent a new layer, and a brand cannot remove one. The constraint is the feature: every page composes from the same five parts, so primitives compose coherently everywhere.
Layout variants

Layout — every page is wrapped.

Three named Layout styles ship in Lingua v0.2. default for marketing surfaces (topnav only, generous gutters). app for product/admin/dashboards (topnav + sidenav, denser). bare for auth, embeds, print (no chrome, content fills the viewport). Each card below is a mini-mockup of the chrome the variant produces — the boxes annotated map to the wrapper keys the JS theme will populate.

layout.styles[0] · marketing

default

TopNav present, SideNav absent. Generous content area centered to 1280px. Used by homepage, comparison page, theory page.

topnav links + cta
topnav: full sidenav: none
layout.styles[1] · product

app

TopNav + SideNav. Denser gutters, narrower content column. Used by site overview, page editor, dataset view, civic dashboard.

topnav env + user
topnav: full sidenav: rail
layout.styles[2] · auth, embed

bare

No TopNav, no SideNav. Content fills the viewport. Used by sign-in, sign-up, embedded views, print.

topnav: none sidenav: none
wrapper keys → Each Layout style sets five class strings: outerWrapper (page background), wrapper (full-page flex direction), wrapper2 (topnav-+-content column), wrapper3 (sidenav-+-content row), childWrapper (parent of the LayoutGroups). The brand styles each — the translation skill will map these to layout.styles[N].* in theme.js.
LayoutGroup variants

LayoutGroup — the band recipes.

A LayoutGroup is one band of content within a page. Five named styles ship in Lingua v0.2. Each card below shows the surface recipe — box vs unbox, shadow tier, padding density, background. The dark band is new in v0.2: it lets a page drop into the code surface for a code hero, terminal demo, or dark CTA without leaving the platform's structural grammar.

layoutGroup.styles[0]

content

The default body band. Boxed surface, 8px corners, soft --shadow-sm, generous interior padding. Holds the main Sections of any page.

layoutGroup.styles[1]

header

The hero band. Unboxed — content sits directly on the Layout surface. No shadow, no rounded box, generous vertical padding. Page title + lede + CTAs go here.

header · unboxed
no box no shadow vertical pad
layoutGroup.styles[2]

auth

Centered vertically and horizontally. Narrow boxed surface (~440px). 12px corners, --shadow-md. Pairs with the bare Layout for sign-in / sign-up.

layoutGroup.styles[3]

footer

Full-bleed band at page bottom. Sunken background, top border rule, no shadow. Holds the link block + brand + meta on every page.

layoutGroup.styles[4] · new in v0.2

dark — the code-surface band

Full-bleed band on the brand's deep slate code surface (--color-code-bg, #0E0F12). Used for code heroes, terminal demos, architecture-diagram bands, and the occasional dark CTA. The light-to-dark shift between sections is Lingua's compositional fingerprint — budget at least one per marketing page.

dark · code surface · full bleed
code surface brand signature full bleed no shadow
wrapper keys → Each LayoutGroup style sets three class strings: wrapper1 (outer full-width band — sets background + bleed), wrapper2 (inner container — max-width + padding), wrapper3 (content alignment). The translation skill maps these to layoutGroup.styles[N].* in theme.js.
v0.2 add → The dark variant is new. v0.1 had no dark band; pages dropped to dark by overriding wrapper classes inline. v0.2 makes the dark drop a first-class variant so authors can pick it from the band-style dropdown without editing CSS.
Nesting example

A Layout with five LayoutGroups stacked.

Below: one Layout (the marketing default variant) containing a header band, two content bands, a dark code band, and a footer band. The frame guides mark which wrapper key each region maps to. Sections live inside each band — the dark band would typically hold a code-block Section and a callout, not bare lines.

read this as → the order of LayoutGroups is the order they render. An author drops a dark band between two content bands by picking it from the band-style dropdown — no code, no override. The grammar lets the visual energy of a page shift mid-scroll without leaving the platform.
The page as a row

Every page above — pages, sections, the lot — is a typed row.

Lingua's lingua franca is the row format. A Layout is a row. A LayoutGroup is a row. A Section is a row. The page you're reading lives in the database as something close to the JSON below — one parent row, child rows for each band, grandchild rows for each Section. The platform reads them, the renderer composes them, the theme styles them.

design-system/layouts.row.js 
// this very page, expressed as DMS rows
export const layoutsPage = {
  app:    "lingua",
  type:   "design_system|page",
  parent: "lingua|design_system:pattern",
  data: {
    title:  "Layouts",
    slug:   "layouts",
    layout: "default",      // one of layout.styles
    sections: [
      { group: "header",  component: "Lexical", title: "Layouts — every page is wrapped" },
      { group: "content", component: "Lexical", title: "Hierarchy diagram" },
      { group: "content", component: "Card",    title: "Layout variants", source: "layouts" },
      { group: "content", component: "Card",    title: "LayoutGroup variants", source: "groups" },
      { group: "content", component: "Lexical", title: "Nesting example" },
      { group: "dark",    component: "Lexical", title: "The page as a row" },
      { group: "content", component: "Table",   title: "Naming reference", source: "naming" },
      { group: "footer",  component: "Footer" },
    ],
  },
};

// every row, everywhere, has the same shape:
// { app, type, data, parent, children }
why this matters → Because the page is a row, it can be edited from the admin UI, exported as JSON, version-controlled, diffed, and re-imported on another site. The author touches the same shape the engineer touches. One vocabulary. Many speakers.
Naming reference

Which Layout, which bands — a quick reference.

When a brand or author asks "what should this page use?" — this is the table they read. Lingua ships three Layouts and five LayoutGroup variants; almost every page in the platform is composed from a small number of recipes below. New surfaces should match one of these unless the design genuinely needs a new variant (and if it does, add the variant to layouts.html before using it).

Page kind Layout Bands (in order) Notes
Marketing homepage default header · content · dark · content · footer Hero band + feature grid + code band + customer logos + footer. The dark band carries the brand's contrast move.
Marketing comparison default header · content · dark · footer Hero + side-by-side architecture + capability matrix; dark band for the terminal demo or "speak it" code sample.
Marketing theory (essay) default header · content · footer Long-form prose, narrow measure (720px). Editorial italic accents permitted on this page only. No dark band by default.
Documentation page app content (narrow) · footer SideNav carries the docs tree; content column is narrow (720px). Code blocks are frequent — they sit on the dark surface inline, no dark band needed.
Product site overview app content · content · content KPI strip + patterns table + live activity. Full content density; sidenav is the primary navigation, no marketing header.
Product page editor app content (full bleed) Three-pane editor (section list + canvas + inspector) fills the band. No header band — the topnav carries the page title.
Civic / analytics dashboard app header · content · content · footer Slim header (page title + filters), KPI strip + charts + tables in content bands. Dark band optional for the "what is this" intro.
Sign-in / sign-up bare auth No chrome. One centered card. The auth band brings the rounded surface + shadow; the bare Layout strips the nav rails.
read this as → three Layouts × five LayoutGroup variants is enough to describe every Lingua surface in the catalogue. Authors pick from these dropdowns; brands restyle the variants but don't add new ones — that's how the system stays composable across themes.
what about a new band? → If a surface needs a band that doesn't fit any variant above, add it to layouts.html first — design it as a named LayoutGroup style, give it wrapper keys, then use it. Don't smuggle one-off CSS into a page; that's how systems drift.
