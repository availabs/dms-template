5 · Brand-signature primitives

Code, terminals, architecture, KPI, activity — how Lingua proves its claims.

These are the v0.2 brand's defining surfaces. Light page, dark code block. Boxy rounded architecture diagram. KPI strip in mono. Live activity in a vertical rail. Customer logos in their colors. The brand earns "modern technical" by showing — not by saying.

Code block

The brand's most distinctive surface. Deep slate background, JetBrains Mono, syntax-highlighted, optional window chrome (filename + copy button). At least one per marketing page.

a Full block with chrome · row.js
row.js 
// Every row, everywhere in Lingua, has the same shape.
// A page is a row. A section is a row. A dataset is a row.

export const row = {
  app:        "the_drift",
  type:       "essays|page",
  data:       { title: "A defense of shared shape", wordcount: 3420 },
  attributes: { live: true, publishedAt: "2026-05-24" },
  parent:     "essays|page:vol_14",
  children:   [ "essays|page:vol_14|section:lede" ],
};

// Pages, charts, maps, datasets — all read the same shape.
export function render(row, theme) {
  return theme.components[row.type.split("|").pop()];
}

The lingua franca of data

A page is a row. A section is a row. A dataset is a row. The same typed shape drives every primitive Lingua ships — and because the shape is shared, the parts compose without translation. That's the idea. Below: what it actually looks like when you build something.

One vocabulary, many speakers

The brief was simple. A non-technical author opens the editor and sees pages and sections. An engineer opens the same project and sees typed rows, parents and children, and a Falcor-shaped API they can hit from anywhere. Both views are real; both surfaces are first-class.

What you write

Pages. Sections. Datasets. Themes. All of them rows of { app, type, data, attributes, parent, children }.

  • A marketing page is a page row with section children.
  • A dashboard is a page row with data-section children that read from datasets.
  • A dataset view is a source row with view children that share the same data table.

What you get

  1. One renderer for every surface.
  2. One API for every consumer.
  3. One migration story when the schema changes.

The exception that proves the rule

The exception is the long-form essay. We keep one editorial face — Fraunces italic — for exactly that surface. It's the brand's one moment of publication, deliberately distinct from everything else.

The most successful data tools are the ones whose representation is small, fixed, and universally legible.

What that looks like as code:
