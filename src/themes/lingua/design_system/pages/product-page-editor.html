A field guide to durable systems.

Ten essays on the things software keeps having to relearn — protocols, file formats, plain text, and the long quiet work of writing for the people who come after you.

S. PERETZ11 MIN

The first three essays of the issue circle a single argument: the representations a system commits to are more durable than the code that manipulates them. A team will rewrite its database client three times before it considers changing the on-the-wire format. A protocol that has seven implementations will not be retired, even if every implementation is undermaintained. The shape is sticky; the code is fluid.

The essays in the rest of the issue take this seriously and look at what follows from it. If the format outlives the code, the format deserves the same care that the code gets — versioning, documentation, deprecation plans, ergonomics, taste. The plain-text people figured this out a long time ago. The rest of us are still catching up.
