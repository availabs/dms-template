Themes — the brand the rows wear.

A Lingua theme is a small JavaScript object that maps every primitive in the platform to a brand — the same row data, rendered in your voice.

v0.2 Last edited 2026-05-24 10 min read

Lingua treats every page, section, dataset, and query as a row with the same shape. A theme is the layer that decides what those rows look like on the screen — the font of a headline, the radius of a button, the depth of a shadow under a card, the color of a focused input’s border. The data underneath is invariant; the theme is the dialect spoken on top of it. Two sites can hold the same content and feel completely different, because the theme — not the row — owns the visual voice.

Themes are plain JavaScript. There is no theme compiler, no preprocessing step, no proprietary token format. Everything a theme can change is reachable through three top-level keys: layout, components, and textSettings. The rest of this chapter walks through how those keys compose, what they expect, and how to author a theme that survives the four canonical surfaces — marketing, docs, dashboards, and dataset views — without forking.

Before we open a file: a theme is additive. You don’t inherit the default theme and then override it — you supply only the keys you want to change, and Lingua deep-merges your object over the baseline. This is why most published themes are short. The Lingua default is already opinionated; a brand-correct theme usually fits in under 400 lines.

themes/my-brand/index.js 
// A minimal theme. Three keys; everything else inherits from the default.
import { defaultTheme } from "@availabs/lingua";

export const theme = {
  name: "my-brand",
  version: "0.1.0",

  layout: {
    styles: [
      { name: "default", outerWrapper: "bg-white text-zinc-900" },
      { name: "app",     outerWrapper: "bg-zinc-50 text-zinc-900" },
    ],
  },

  components: {
    button: {
      primary: "bg-[#3B5BFF] text-white rounded-md px-4 py-2",
      ghost:   "text-zinc-700 hover:bg-zinc-100 rounded-md px-3 py-2",
    },
  },

  textSettings: {
    styles: [{
      display: "font-display font-bold tracking-tight",
      prose:   "font-sans text-base leading-relaxed",
    }],
  },
};

A theme that exports an object of this shape is registerable. The next section — Anatomy of a theme override — walks each key, but it’s worth pausing on the structure. layout owns the chrome around every page: outer wrapper, top nav, side nav, the slot where content lives. components owns the visual treatment of every primitive the platform renders — button, card, input, multiSelect, topnav, and the rest. textSettings owns the named font styles authors pick from inside Card cells.

themes/my-brand/layout.js 
// Layout styles are an ordered list; the page tree picks one by name.
export const layout = {
  styles: [
    {
      name: "default",
      outerWrapper: "bg-white text-zinc-900",
      wrapper:      "flex flex-col min-h-screen",
      wrapper2:     "flex-1 flex flex-col w-full",
      wrapper3:     "flex flex-1 w-full",
      childWrapper: "flex-1 flex flex-col",
      topNav:       { size: "md", sticky: true },
      sideNav:      { size: "none" },
    },
    {
      name: "docs",
      outerWrapper: "bg-white text-zinc-900",
      childWrapper: "grid grid-cols-[256px_1fr_220px]",
      topNav:       { size: "md", sticky: true, search: true },
      sideNav:      { size: "compact", side: "left" },
    },
  ],
};

Each Layout style is a recipe for the page chrome. The page’s row decides which recipe to use by name (layoutStyle: "docs"), and Lingua renders the matching wrapper stack with the configured navigation. The recipe you’re reading right now uses a three-column docs layout — a sidebar on the left, a 720-pixel reading column in the middle, an on-this-page rail on the right — assembled from exactly those keys.

The CLI scaffolds the four files at the shape Lingua expects, then registers them in the theme index. If you prefer to start from someone else’s theme, lingua theme fork <name> copies an existing theme into your project so you can edit it like your own. Either way the result is the same: a folder of plain JS the rest of the codebase can read.

Anatomy of a theme override

Most theme work happens inside components. Each primitive entry is either a string of Tailwind classes (for primitives with a single visual treatment) or an object of named styles, where each named style is itself a string of classes. The named-style pattern is what lets a brand ship variants — primary, secondary, danger, ghost — that author-facing tools surface as a dropdown. The contract between theme and platform is the set of keys the primitive’s .theme.jsx sibling declares; supply any subset and the rest fall back to the default.

The button is the canonical example. Its .theme.jsx sibling exports a default object listing every key the live <Button> reads: primary, secondary, ghost, danger, plus optional state variants like disabled. A theme that ships only primary and ghost will still render the other two — the live <Button> just falls back to the default classes for the unsupplied keys.

themes/my-brand/components.js 
// One named style per button variant. Each value is a string of
// Tailwind utilities the live Button.jsx will apply.
export const components = {
  button: {
    primary: [
      "inline-flex items-center gap-2 px-4 py-2",
      "bg-[#3B5BFF] hover:bg-[#2F4AE0] active:bg-[#2540B8]",
      "text-white text-sm font-medium rounded-md",
      "transition-colors duration-150 ease-out",
    ].join(" "),
    ghost:   "text-zinc-700 hover:bg-zinc-100 rounded-md px-3 py-2",
    danger:  "bg-[#FF5630] text-white rounded-md px-4 py-2 hover:bg-[#E04020]",
  },

  card: {
    wrapper: "bg-white border border-zinc-200 rounded-lg shadow-sm",
    header:  "p-6 border-b border-zinc-100",
    body:    "p-6",
  },
};
Note

Theme strings are Tailwind utility classes resolved at build time. If you reach for an arbitrary value — bg-[#3B5BFF] — make sure your tailwind.config.js safelists the brand palette, or the class will be pruned in production. The CLI’s lingua theme apply step rewrites the safelist for you; if you wire a theme by hand, mirror its content array.

The same pattern applies to every primitive: input, multiSelect, tabs, switch, card, topnav, sidenav, dataCard. The list of keys is fixed by the source files; the values are entirely yours. To find the keys for a primitive you haven’t touched before, open its .theme.jsx sibling in src/dms/packages/dms/src/ui/components/ — the default export is the contract.

When a row is rendered, Lingua walks the theme in a fixed order: the site-level theme first, the Layout style’s overrides next, the LayoutGroup style’s overrides after that, then the Section’s, then the Component’s. Each layer can shadow the one above it without replacing it — an app-style Layout that sets a denser card.body doesn’t lose the brand’s card.wrapper radius. The merge is deep and class-aware, so two layers that touch the same key get a sensibly resolved string rather than a stomped one.

This is why themes survive surface changes. Add a new Layout variant tomorrow — an analytics shape with a sticky filter bar — and your existing theme keeps working: the analytics Layout reads the same component keys as every other Layout. The brand voice is decoupled from the page chrome because the merge contract guarantees it.

themes/my-brand/textSettings.js 
// textSettings.styles[0] is the brand's default named-style set.
// Authors pick these by name in the Card cell editor's "Value font" dropdown.
export const textSettings = {
  styles: [{
    displayHero: "font-display font-bold text-7xl tracking-tighter leading-[1.02]",
    displayXL:   "font-display font-semibold text-5xl tracking-tight leading-tight",
    displayMD:   "font-display font-semibold text-3xl tracking-tight",
    h2:          "font-display font-semibold text-2xl tracking-[-0.012em]",
    h3:          "font-display font-semibold text-xl tracking-[-0.008em]",
    prose:       "font-sans text-base leading-relaxed text-zinc-900",
    proseSM:     "font-sans text-sm leading-normal text-zinc-700",
    mono:        "font-mono text-sm tabular-nums text-zinc-700",
    monoXS:      "font-mono text-[11px] uppercase tracking-[0.06em] text-zinc-500",
  }],
};

The textSettings.styles[0] object is the type system in code form — one entry per token in your design system’s Type section. Add an entry here and it appears in the Card cell editor’s “Value font” dropdown, available to every author across the site. Remove one and the pages that referenced it fall back to prose. This is how a brand keeps its type discipline portable: the dropdown is the entire menu.

Themes are also the right place to register brand-specific column types — small focused renderers that the Card grid can slot in alongside data cells. The theme.columnTypes map carries them; each entry pairs a cardHints object (so the Card knows the cell’s default span and behavior) with a ViewComp React component. The pattern is intentionally narrow: one column type renders one visual element. Anything larger should be configured Card cells, not a custom component — see Card columns for the longer argument.
