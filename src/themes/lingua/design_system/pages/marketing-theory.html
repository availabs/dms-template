essay · on representation · Theory No. 01

On placing.

Most stacks for data-driven sites translate the same underlying data into four or five different representations — one for the CMS, one for the dashboard, one for the static site, one for the analytics — and pay the cost of that translation forever. Lingua chooses the opposite: one typed row, rendered many ways.

By Alex Richter · 2026-05-24 · 14 min read · Theory
01 · The translation problem

The translation problem.

There is a default stack for building data-driven sites, and almost everyone reading this has built one. A CMS holds the pages. A spreadsheet or a Postgres table holds the data. A dashboard tool reads the data and renders charts. A static-site generator builds the public pages and bakes the charts in as images, or iframes them in from the dashboard tool. An analytics tool sits on top of all of it, instrumenting the rest. The result is shipped, and it works, and the team moves on.

The cost shows up later, and it is always the same cost. The same underlying datum — a row about a building, a row about a sensor, a row about a member — lives in four or five different shapes across the stack. Each tool was built around its own representation, and each integration is a translation layer between those representations. When the row changes, the translation layers change. When the team adds a new view of the data — a map, say, or a different chart cut — the translation has to be written again, against a new tool that speaks a sixth shape.

This is the situation Tom Macwright is pointing at when he writes that the energy in programming goes into syntax, control flow, and connecting things, but not nearly enough into what those things are[1]. His one good example is Excel: Excel is powerful, he argues, not because formulas are simple, but because it has only sheets, columns, rows, and cells. The leverage is the datatype, not the language. Once you and the spreadsheet agree that a cell is the unit, every formula, every chart, every conditional format, and every external integration composes against that one shape. There is no translation layer because there is nothing to translate to.

The same point holds for pandas in the Python data ecosystem. Seaborn, scikit-learn, statsmodels — none of them sat down and negotiated a shared interface. They all just took the dataframe as given and composed against it. The dataframe is the lingua franca; the ecosystem is the consequence.

The leverage is the datatype, not the language.

after Macwright, 2021

What both examples have in common is the absence of a translation layer between the unit of data and the things that act on it. A chart in Excel does not translate the cell into a different model and then render the model; it queries the cells and draws. A scatter plot in seaborn does not unpack the dataframe into a graph IR and then convert; it iterates the rows and plots them. The work that would otherwise go into translation goes instead into rendering — into the act of actually showing the data — and the gap between “the data” and “the picture” collapses.

Sites for data — the kind of work Lingua is built for — have not yet had this collapse. The page is one representation, the chart is another, the table is a third, the map is a fourth, and the analytics about all of them is a fifth. The team that maintains the site keeps the five representations in sync by hand, or with a fragile pipeline, or by paying a vendor whose entire business model is being the glue. None of those answers is wrong, exactly; but all of them concede that the translation is unavoidable. The argument of this essay is that it isn’t.

02 · The representation

What “one row” means.

Lingua has exactly one datatype. It is a typed row in a table called data_items. Every page is a row in this table. Every section on a page is a row. Every dataset is a row. Every view of a dataset is a row. Every column configuration on that view is a row. Every theme, every plugin manifest, every authored fragment of rich text — rows. The same shape, with the same five-or-six top-level fields, distinguished only by a type string that tells the system how to render this particular instance.

The shape is small enough to fit in a paragraph. A row has an app (which site it belongs to), a type (what kind of row it is), a parent (what it nests under), a data blob (the row’s actual content), and an optional attributes map for metadata. That’s it. There is no “page object” with a different schema from “dataset object”; there is no separate “theme object” with its own table. The same row format describes a marketing homepage, a hundred-thousand-row sensor dataset, the theme that styles them, and the analytics that count them.

It is reasonable to be suspicious of a system whose first claim is “everything is a row.” That phrasing is close enough to the phrasing of every “infinitely flexible” tool of the past decade to make a careful reader stop reading. The right way to hear the claim is comparatively: this is what Excel has always done with cells, and what pandas has always done with dataframes, and what SQL has always done with relations — it is what made each of those systems durable. The novelty in Lingua is not the row; the row is the conservative move. The novelty is applying representation primacy to the part of the stack — site, content, page, theme — that has historically refused it.

The pragmatic consequence is that the boundary between “CMS” and “data platform” collapses. A page is just a row whose type ends in :page. A section on the page is a child row whose type ends in |component. The dataset that the section reads from is another row whose type ends in :source. A theme is yet another row. A query is a row. You can write a Postgres query against data_items that finds every row of every kind, across the whole site, because there is no other table to look in.[2]

An implementation note

Rows that grow very large — a sensor dataset with ten million readings — are auto-routed to per-type split tables so the index on the main table stays cheap. The split is invisible above the storage layer; the query API still sees one shape. Detection is a string suffix on the row’s type field, which is the closest thing in the system to a special case.

It is worth being precise about what this is not. It is not “a metadata layer.” It is not “a graph database” or “a triple store.” Those are systems where the schema is open and you commit to defining your own. Lingua is the opposite: the schema is closed, the row shape is fixed, and the openness is in how each row is rendered. The constraint is the feature, exactly as it was for Excel and the dataframe; the constraint is what lets pieces of the system that were written separately compose without translation.

03 · The implications

On placing.

Once the row is the unit, several things stop being arguments. The first is the “author versus engineer” argument. In the default stack, the author works in the CMS UI and the engineer works in the dashboard code, and the two communicate by handing off requirements. In Lingua, both audiences look at the same row store. The author sees the rows through a configured Card section; the engineer sees them through a query, or a CLI command, or a direct table read. There is one underlying surface, addressed in two idioms. A request like “add a column to this view” is the same operation regardless of who runs it.[3]

The second argument that stops is the “static versus dynamic” one. Because every chart is a query and every query is a row and every row is materially the same kind of object, a page can include a chart by referencing the dataset row in the same way a static section references a content row. There is no separate “data pipeline” standing between the page and the data; the page and the data live in the same table. A change to the data is visible to the page on the next render. A change to the page is visible to the data tooling on the next refresh. The synchronization problem — the thing the typical stack solves with cache-invalidation rituals — doesn’t arise in this form.

Authors and engineers see the same surface, addressed in two idioms.

The third argument is the one about themes. In the default stack, a theme is a set of overrides applied to a CMS render; the dashboard has its own visual conventions; the static site has a third set of conventions; the map widget’s embed has a fourth. The brand is approximately consistent because someone has gone to the trouble of producing four parallel style libraries. In Lingua, a theme is a row that contains class-string overrides for every primitive in the system. The same theme styles the marketing homepage, the documentation page, the dataset view, the page editor, the civic dashboard, and the embedded map. Authors don’t pick a brand for “the site” and another for “the dashboard”; they pick a theme for the row store and every render inherits it.

None of this should be read as a pitch for monoculture. A theme is one row; a site can have several. A column type is a plugin; a site can register many. The point is not that there should be one of everything — the point is that the things that are many should be pluggable against a shape that is one. This is the move that makes the WordPress.org / .com distinction—an open-source core, plus a managed hosting offering on top—coherent rather than confusing[4]: there is one underlying representation, and the hosted product and the self-host product are two ways of running the same engine. The portability that follows is not a marketing claim; it is an artifact of the shape.

What the row buys, in the end, is a particular kind of placing. The page is placed inside the same store as the dataset that fills it; the chart is placed alongside the rows it visualizes; the theme is placed beside the surfaces it styles; the author is placed beside the engineer. Each piece sits next to the others in the same vocabulary, and the work of joining them — which in the default stack consumes most of the energy — becomes a small, ordinary lookup. The space that energy used to fill is freed for the things that actually distinguish a site: the writing, the data, the argument it is trying to make.

Notes

  1. Tom Macwright, “One way to represent things,” macwright.com, 23 Feb 2021. The essay’s argument about representation primacy — that powerful systems are powerful because they constrain to one good representation — is the closest intellectual antecedent for Lingua’s row format.
  2. The actual query is SELECT * FROM data_items WHERE app = $1. Filtering by row kind is a LIKE on the type column — for example, type LIKE '%:page' for every page in a site. The full type grammar is documented in the DMS Library reference, under “Data Model.”
  3. The Card section — the workhorse rendering primitive — is configured by picking columns from a source, setting cell spans, and choosing styles from the theme’s textSettings. The same operations are available through the admin UI and through the row JSON; an engineer editing the JSON and an author editing the UI are writing into the same field.
  4. The WordPress.org / .com split is the canonical reference for “same engine, two distributions.” See Matt Mullenweg, “.org / .com: a clarification,” wordpress.org/news, 2014 — though the operative distinction was already present in the founding split a decade earlier.
  5. The phrase “representation primacy” is borrowed informally from the type-theory literature, where it has a narrower technical meaning; the usage here is the looser one Macwright introduces. See also Richard O’Keefe, “you cannot get a simple system by adding simplicity to a complex system” — quoted by Macwright in “A clean start for the web,” Aug 2020.
