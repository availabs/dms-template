Unified tree — one surface replaces the v1 Pages / Publish Queue / Site Tree / Navigation tabs. Built as a custom SiteTreeSection component (the v1 Site Tree annotation already concluded a Spreadsheet can't express hierarchy): full pages load, tree assembled client-side by child.parent === parent.id, siblings ordered by index. Note: the real mitigateny_2025 data stores parent as a page id, not a title — and with "Risk Profile" ×12 and "Mitigation Strategy" ×13 in the same pattern, any title-based join (as sketched in the v1 page_children annotations) would mis-count. Hierarchy columns must join on id. Pagination off — same full-load constraint planning/tasks/current/page-hierarchy-columns.md already accepts.

Lenses = saved filters, not tabs. To Publish ≡ published='draft' OR has_changes=true (v1's Publish Queue); Empty ≡ section_count=0; Orphans ≡ parent id not found; Off Nav ≡ hide_in_nav=true. A lens filters the tree but keeps ancestors visible (dimmed) so a flagged page is always seen in its place in the site. Real-data shape these lenses serve: 30 changed + 2 drafts, 44 empty pages (incl. ~40 "ghost" rows with no data at all), 3 orphans, 1 hidden page. The publish queue is overwhelmingly changed pages, not drafts — so the State column shows Published + an amber changes-dot rather than leading with a Draft pill.

Navigation folded into the tree — sibling order is nav order (index), so a separate Navigation tab becomes redundant: the ⠿ handle reorders siblings (drag deferred — needs the bulk-update primitive; ↑↓ swap of adjacent index works today, task site-mgmt-navigation-management.md), and the In Nav toggle writes hide_in_nav via apiUpdate inline. Published and nav visibility stay independent, as in v1.

Sections as the tree's second level. Clicking a page's section chip expands an inline sub-table of that page's {pattern}|component rows (component_type / source_id / view_id virtual SQL columns via extractDmsJsonPath, element_preview modal — v1's Sections tab, scoped to where the section lives). Real pages average 15 sections and run up to 59, and most are untitled lexical body sections — only the titled, leveled ones form the page outline. Hence the panel's Outline / All toggle: Outline shows the titled skeleton (what an editor scans), All shows every row in a scrolling list. The Sections search scope answers cross-cutting questions by filtering sections globally and auto-expanding matches under their pages. Its source dropdown doubles as a data-source inventory: "Any data source (N)" shows only sections bound to a source, and each option carries its site-wide section count, grouped Internal (DMS) vs External (UDA). Both kinds come from the same place — the section's element-data.sourceInfo object (shape verified on live Flooding sections: { name, source_id, env, srcEnv, view_id, view_name, updated_at }); internal DMS sources have srcEnv "mitigat-ny-prod+prod", external UDA sources env "hazmit_dama". The earlier v1 sketch only read externalSource — the real extraction must read sourceInfo to catch both. With a type/source filter active, panels show matching rows only (the page row above is the context), a binding-by-binding audit pairing with the Data Sources surface's "Used by" counts.

View staleness & bulk re-binding. Each source-bound row shows its View next to the Source: sourceInfo.view_id vs the source's newest view (max view id over {source}|{name}:view rows for internal sources; the UDA views list for external ones). Live example: Flooding's event sections are bound to AVAIL - Fusion Events V2 (12/10/2024), view 1648 — when a newer view is published, the amber v1648→1700 chip appears, the page's section-count chip gets an orange dot, and the Stale Views (N) filter collects every lagging binding site-wide. Per-chip click rebinds one section; Update All to Latest Views batch-rewrites sourceInfo.view_id/view_name across matches (needs the same batch-updateItem primitive as bulk publish). Updated sections flag v1700 ✓ for column-mapping review — a new view can drop or rename columns the section's dataRequest/columns config references, so re-binding marks the section (and its page's has_changes) rather than silently publishing.

lexicalSummary() — derived content summaries. Untitled lexical rows show an auto summary instead of "(untitled)": walk element['element-data'].text.root.children (shape verified against the live Flooding page, 1009741); return the first heading node's text if present, else the first non-empty text run truncated to ~90 chars, else flag (empty). Styling distinguishes provenance: authored titles bold ink, derived heading semibold, derived excerpt regular with a ¶ mark, empty pale italic. Summaries are searchable in the Sections scope (full-text-ish search over page content from site management). The Flooding panel in this mockup carries the real extracted summaries — including 6 truly empty lexical sections it exposed (cleanup candidates, flagged in the panel header). Implementation: client-side helper next to the section sub-table (the recursive walk over Lexical JSON doesn't fit a SQL virtual column); if the full-load walk gets heavy at 2,478 sections, persist a content_summary attribute computed on section save.

Bulk publish — select-rows + batch updateItem is a new primitive (same dependency v1 flagged). Per-row Publish / Discard reuse page_publish behavior (editFunctions.jsx: set published='', has_changes=false, append history entry). Inspector (click a page title) reads data.history.entries — same source as site-mgmt-last-published-column.md. Data Sources / Site Health / Activity remain separate surfaces, unchanged from v1 — though the tree already absorbs the health checks that are structural (empty pages, orphans) as lenses.

Reference data — structure, counts, names, and pathologies in this mockup are drawn from the live prod|mitigateny_2025:pattern (app mitigat-ny-prod, pattern 985070): 164 pages / 2,478 sections; Natural Hazards with 16 hazard children each carrying a Risk Profile (41–59 sections) and Mitigation Strategy (17–21); Mitigation Matters with 21 flat story pages; ghost rows and orphaned test pages. Dump: scratchpad/mitigat-ny-prod-prod/mny2025-pages-full.json.
