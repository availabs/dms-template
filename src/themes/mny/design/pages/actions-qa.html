The verdict

The map is nearly complete.
It still can't tell you where anything is.

The first audit of this dataset found that 38% of mitigation actions had no point at all — they simply could not be drawn on a map. That was a code problem, not a data problem: three of the four rungs in the location waterfall were broken by type mismatches, a geoid stored as a JSON array here, a number there, a coordinate field nobody read. Those bugs are fixed, and the dataset was re-published on 14 July 2026.

Coverage went from 61.9% to 96.7%. Six thousand three hundred and eighty-five actions that had been invisible now appear. New York City, which had not a single mappable action out of 2,082, is fully placed. Only 373 actions remain genuinely unresolvable.

And the map is no more accurate than it was. Every one of those recovered points is a centroid — the geographic middle of a town, or worse, of an entire county. Ninety-nine point six percent of the points on this map describe a municipality, not a project. Seventy-four actions out of 18,378 sit where the work will actually happen.

That is the thing to hold onto. Coverage was a bug, and bugs get fixed. Precision is not a bug — the actions dataset simply does not record where the projects are, and no amount of code will conjure it.

The Precision Ladder, Repaired

Each action is pushed through the rungs in order and stops at the first one that yields a point. The rung it stopped on is its precision score. Bars are the current dataset (view 12463); the tag on each row is what the same rung produced before the fix.

The rung now reads geometry_lat_long_polygon_etc, the free-text field that actually holds coordinates, instead of longitude/latitude, which do not exist here. A ReferenceError that would have crashed the entire publish task the moment it found a coordinate is also gone. These 46 points, plus the 28 geocoded ones, are the only real locations in the dataset.

Jurisdiction centroid

Middle of the town · median error 5.1 km

Improved

Still the rung that carries the dataset. Coercing every geoid to a string at the cache boundary recovered the 566 actions whose geoid_juris was a JSON number and so missed a string-keyed lookup. 552 actions still have a geoid_juris that matches no jurisdiction — several are 5-digit county FIPS sitting in the jurisdiction field. They fall to a county centroid; they should resolve to a town.

The safety net that had never caught anything now catches 90% of everything the fix recovered. The bug was that county_geoid is a one-element JSON array — ["36091"] — matched against a string-keyed map, so the comparison was never true. Unwrapping it works. But read the error figure again: these 5,775 actions are now on the map, each one a median 26 km from where the work is. They are visible, not located.

What is left is genuinely unplaceable: 356 of the 373 have no county, no jurisdiction and no geoid of any kind. Sixteen have no county geoid, and one carries the literal string #ERROR_#N/A. No code change will place these — only new data will.

How wrong is a centroid?

A centroid answers "which municipality" — never "where". Measured against the real geometry of every New York jurisdiction and county, this is how far a project can sit from the point that represents it.

Before the fix, almost every mapped action was a town centroid. Now one in three is a county centroid — five times coarser. Coverage rose and the typical error rose with it. Surface the precision code wherever these points are drawn.

county_geoid was an array

Fixed

97% of rows stored the county as ["36091"], a one-element array, passed straight into a map keyed by string. The comparison was by object identity and never true — the rung fired once in 18,378 rows. It now unwraps the array and walks every element of a multi-county action.

Fixed

Rung 1 was a landmine

Defused

Besides reading fields that don't exist, the coordinate helper had a typo that threw ReferenceError the moment it did find a coordinate. It was dormant only because the fields it looked for were absent — one latitude column away from killing the whole publish task.

Prevention

The old worker logged only final counts, which looked plausible. It now prints candidates versus hits for every rung — a rung with 5,775 candidates and one hit is visibly broken, not merely sparse. Every one of these bugs would have been obvious on day one from four lines of log.

The dataset has two halves that fail in opposite directions. The location fix closed the gap between them; the completeness gap is untouched, because it lives in the actions data itself and no pipeline change can reach it.

Statewide actions are narrative-complete and place-blind. Every one has a name, a lead agency and a funding source. Almost none has a cost range (3.3%) or a jurisdiction geoid (10.5%).

Most of them are now honestly labelled rather than falsely placed: 235 carry precision 5, "statewide — no point". But it turned out not all state actions are placeless. Thirty-three are New York City actions and eighteen name a specific borough — "Sunset Cove Restoration Project" in Queens is a real project that happens to be funded through the state plan. Those keep their point.

Local actions are the mirror image — 98% have a county and 82% a cost range. They are now 97.9% mapped. And curiously, statewide actions remain the only group with real coordinates: 11.6% populate the geometry field, versus 0.7% of local ones.

Still true: do not count statewide actions in map-coverage metrics. A placeless action isn't a coverage failure — treating it as one creates pressure to invent a point for it.

action_status is the clearest case. It is populated on 99.9% of rows — and 91% of those rows say the same thing, Proposed – Not Started. Any dashboard that slices by status will draw one bar. The field is complete and almost entirely uninformative.

The cost fields tell a similar story from the other direction: estimated_cost reads 53.9% complete, but 1,178 of those entries are the literal strings TBD, N/A or unknown. The true rate of usable cost data is closer to 47%.

Note the last two rows. 1.3% of actions have an address and 0.9% have coordinates. That is the entire raw material for a precise map — and it is why the precision number cannot be moved by code.

The thing to remember

Coverage went from 61.9% to 96.7% and 99.6% of the points are still centroids. The dashboard number now looks solved while the map remains an approximation — which is exactly the moment a quality metric becomes dangerous. Do not report "% of actions mapped". Report the precision distribution, or the number will flatter a map that still cannot tell you where anything is.
