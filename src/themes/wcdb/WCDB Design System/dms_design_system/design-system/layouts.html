Foundations · layouts · v0.1

The cutaway, and its grammar.

WCDB ships one Layout — the two-column cutaway split where a sticky hero panel on the left sits next to a scrolling editorial column on the right. The Layout exposes two named LayoutGroup variants — content and header — that the cutaway requires. This page is the literal reference for both.

01 — Hierarchy

Site → Layout → LayoutGroup → Section → Component.

Every WCDB page renders as a stack of these five primitives. Layouts and LayoutGroups are invisible chrome — they don't render until a Section asks for space inside them. The hierarchy below is what the runtime mounts when a reviewer visits /home. 

Site                              app="wcdb"  type="prod:site"
└── Theme                         wcdb_theme.js
    └── Layout                    default · the cutaway split
        ├── TopNav                wcdb · compact · floats over header panel
        │
        ├── LayoutGroup           header   (left column · sticky cutaway)
        │   └── Section           card · now-playing strip · height="fill"
        │       └── Component     Card · cells: portrait_banner, stream_player,
        │                                       now_indicator, title, dj
        │
        └── LayoutGroup           content  (right column · scrolling feed)
            ├── Section           lexical · "Tonight on 90.9"
            ├── Section           card · upcoming shows · size="12"
            ├── Section           card · recent spins · size="12"
            └── Section           lexical · footer disclaimer

The cutaway requires both LayoutGroups. A page that ships only a content group (the design-system documentation pages, for instance) renders as a single scrolling column because the md:grid-cols-2 grid only has one child. The class string is the same; the geometry follows the row count.

03 — LayoutGroup variants

Two surface recipes.

WCDB ships two LayoutGroup styles. Each owns its column in the cutaway. The header recipe produces the sticky, rounded, card-bg panel that holds the hero artwork and the now-playing strip. The content recipe produces the unboxed scrolling column that hosts the editorial feed.

content

Right column · scrolling feed.

Sits on var(--page-bg). No card chrome on the group itself — individual Sections inside it carry their own rounded surfaces. Padding is p-2 on the outer ring and p-4 on the inner so Sections breathe inside the column.

04 — Wrapper class reference

The literal strings, verbatim.

Variant cards above show shape; this table is the source of truth for the class strings on each LayoutGroup style. Pulled verbatim from wcdb_theme.js · layoutGroup.styles[0..1]. If these strings drift from theme.js this table is wrong.

Variant wrapper1 wrapper2 wrapper3 Alignment
content w-full flex-1 flex flex-row p-2 flex flex-1 w-full flex-col relative text-[color:var(--ink-1)] text-md font-light leading-7 p-4 min-h-[200px] (empty) column-aligned · no cap
header w-full p-2 md:sticky md:top-0 md:h-screen overflow-hidden rounded-[18px] bg-[var(--card-bg)] text-[color:var(--ink-1)] md:h-full md:h-full column-aligned · sticky · boxed
Callout · why no mr-auto / mx-auto

The skill's wrapper-class reference table normally documents an mr-auto vs mx-auto alignment decision on each variant's wrapper2 — brands that ship product surfaces with a SideNav use max-w-[…] mr-auto so bands hug the sidebar.

WCDB's content group uses flex flex-1 w-full flex-col — no mr-auto, no max-w-* cap on the group. The cutaway Layout already constrains horizontal width to one of the two grid columns at md+ — there is no excess viewport width to consume, so the auto-margin discussion does not apply. The sectionArray inside applies its own max-w-[1280px] mr-auto cap to its grid — see grid.html.

05 — Section width

Regular vs full-width groups.

Inside any LayoutGroup, the sectionArray applies one of two layout classes to its grid container, picked by the editor flag group.full_width. Authors flip this from the Sections pane in edit mode.

group.full_width Class applied Result on WCDB
'off' (default) layouts.centered · max-w-[1280px] mr-auto Sections capped at 1280px, left-aligned within their LayoutGroup column. The default for every editorial Section.
'show' layouts.fullwidth · '' (no cap) Sections fill the parent. On WCDB this means filling one column of the cutaway split — not the full page width. Used by the now-playing card in the header group so the panel reaches the rounded corners.
WCDB constraint

"Full width" on WCDB always means full column width, not full page width. The Layout's childWrapper declares md:grid-cols-2, so each LayoutGroup occupies exactly half the viewport at md+. A full_width: 'show' Section in the right column fills its column — it does not punch through into the left column. There is no workbench variant; a page that needs an edge-to-edge canvas should be authored on a different brand.

07 — Naming reference

Default, app, bare — mapped honestly.

The DMS skill's conventional Layout names (default, app, bare) describe the shape of a brand's chrome surfaces. WCDB collapses the three into one because the brand is the cutaway — there is no app-style SideNav surface, and there is no minimum-chrome login or marketing shell that does anything different.

Convention name Typical role WCDB mapping
default Editorial / content pages. TopNav, no SideNav. layout.styles[0] — the cutaway split. Every page renders here.
app Product / workbench pages. Persistent SideNav. Not shipped. WCDB has no SideNav-bearing product surface — the brand is entirely editorial.
bare Minimum-chrome pages. Login, marketing, errors. Not shipped. The login page uses the same cutaway Layout — the brand's identity is the chrome.

The honesty matters. A theme that ships three nearly-identical Layouts to satisfy the naming convention adds maintenance surface without adding expressive power. WCDB ships one, names it default, and uses it everywhere.

08 — Section heights

Six presets, one sentinel.

WCDB overrides the DMS default pages.section theme to ship a brand-specific heights preset map. Authors pick a height per-section from the Layout → Height control in the section menu.

Preset Value When
autoautoDefault. Content sizes the Section.
fillfill (sentinel)Triggers flex: 1 1 auto. Used by the now-playing card so it expands to fill the sticky header panel.
herocalc(100vh - 160px)Single-viewport hero band. Tuned to the topnav so no overflow.
tall640pxLong card grids — full-week schedule, blog index.
medium400pxStandard editorial Section.
small240pxCompact strips — on-air pill, contact band.
editMinHeight40pxEdit-mode floor so empty sections expose their settings handle. View mode unaffected.

The fill sentinel is what makes the cutaway work. The header LayoutGroup gives its child Sections a flex/grid context with md:h-screen; setting the now-playing card to fill lets it occupy the entire sticky panel from the top of the hero image down to the player strip without hard-coding a pixel height that would drift on every viewport.
