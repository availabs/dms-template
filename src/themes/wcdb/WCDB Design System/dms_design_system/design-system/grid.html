Foundations · grid · v0.1

Twelve columns inside two.

This page documents both grids on a WCDB page: the cutaway layout grid — the md:grid-cols-2 split on childWrapper — and the sectionArray column grid that lays Sections out inside each LayoutGroup. The two nest. Page chrome lives on the outer grid; editorial content lives on the inner one.

01 — The grid spec

Twelve columns, 1280-cap, p-2.

The sectionArray's settings apply inside every LayoutGroup on every page. WCDB recommends the modern 12-column grid; the codebase ships a 6-column default and both are shown for portability.

Tighter sectionPadding (p-2 vs the DMS default p-4) gives the LayoutGroup's own p-4 wrapper2 room to breathe without compounding padding inside narrow columns. The 1280px cap is wider than the DMS default 1020px because at md+ each column of the cutaway only gets half the viewport — the cap has to allow for wide displays.

04 — Row-spans

Rare, but documented.

WCDB's home and listen pages don't use rowspan on Sections — the now-playing card uses heights: fill + flex on the header LayoutGroup to occupy the sticky column. rowspan is available to authors and interacts with the section's heights preset as follows.

05 — Layout-level grid · advanced

Two grids, nested.

WCDB's signature is a higher-level layout grid on the Layout's childWrapper — md:grid md:grid-cols-2. It splits the page into two equal columns at md+, one per LayoutGroup. Inside each column the ordinary sectionArray 12-col grid still applies. The home page therefore has two nested grids: the layout-level split, and a per-column sectionArray.

Each LayoutGroup hosts its own sectionArray. The header group typically hosts a single Section spanning all 12 columns of its narrow half-viewport column; the content group hosts many.

Why nest grids

A single 12-col grid across the whole page couldn't produce the sticky-vs-scrolling behaviour the cutaway requires — sticky positioning needs a flex/grid parent with a defined block axis, and that parent has to be one of the two grid columns, not a sub-span inside a unified grid. The layout-level grid is where sticky lives; the sectionArray grid is where Section size keys live. Authors pick spans inside their column without thinking about the outer split.

The home page is the worked example — see pages/home.html for the rendered output.

06 — In-editor grid view

What "show grid" looks like.

In edit mode the sectionArray draws a gridviewGrid overlay when the author toggles "show grid" on the section pane. The overlay is twelve dashed column lines drawn on top of the content area at the same gutter as the sectionArray itself, so an author can see whether a Section's size key snaps to the intended column.

07 — Rules for picking spans

Default to twelve.

  • Default to size: 12 for every Section inside either column of the cutaway. The Layout has already constrained horizontal width to one half-viewport column at md+ — sub-12 spans are usually too narrow to be legible.
  • Reserve 6 + 6 and 4 + 4 + 4 for the rare case where two or three short Sections share a row inside the right column — a tonight/ tomorrow pair, a three-DJ feature strip. Asymmetric splits (3 + 6 + 3, 8 + 4) should be data-driven — only when the data justifies the asymmetry, not as a stylistic flourish.
  • Row-spans only on the home header group. Use heights: fill on the now-playing Section so it occupies the sticky panel; other pages should leave rowspan at its default of 1.
  • If you reach for an arbitrary width declaration on a Section, stop — that's a layout decision that belongs in the grid's vocabulary. Either pick a documented size key, or surface the missing primitive back to the theme so every future author can express the same thing.
