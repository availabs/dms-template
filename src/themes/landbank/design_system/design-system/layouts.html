Structure · Page chrome

Three layouts, eight bands.

This page documents the invisible chrome: the Layout variants that wrap every page and the LayoutGroup bands that stack inside them. No component here has visible design of its own — layouts and groups are the surfaces everything else sits on, and their wrapper class strings are the spec.

default · public app · admin bare · auth 8 LayoutGroup variants

The hierarchy

Every DMS page is five nested layers. The outer two — Layout and LayoutGroup — are this page's subject: pure structure, no visible design of their own beyond a surface fill and a width decision. Sections lay on the grid (documented on grid.html); components carry all the visible design (components.html).

Site → Layout → LayoutGroup → Section → Component
Site — one DMS site, every pattern and page
Layout — the page chrome · default / app / bare
LayoutGroup — a full-width band; wrapper2 sets width & alignment
Section — a col-span-* p-3 cell on the 12-column grid
Component

Card · Graph · Spreadsheet · Map · Lexical — the only layer with visible design. Everything above it is chrome.

Authors compose the inner three layers in the page editor; developers ship the outer two as named theme variants. This page is the contract for those two.

Why mx-auto here when the platform rule says mr-auto? The platform's mr-auto rule exists so app-surface bands hug a SideNav instead of drifting toward the middle of a wide monitor. The public default layout has no SideNav — it is a civic website, and a civic website centers. So every public band uses mx-auto w-full max-w-[1240px] px-6: an intentional, documented deviation, not drift. The app layout follows the platform rule exactly — mr-auto w-full max-w-[1480px] pl-12 pr-8, the asymmetric padding giving breathing room off the SideNav's edge while mr-auto sends excess viewport to the right. Only auth is centered on both layouts (mx-auto max-w-md); sign-in forms are the one intentionally narrow centered surface.

Section width — regular vs full-width

Inside any LayoutGroup, the sectionArray picks one of two layout classes for its grid container from the group.full_width editor flag. Authors flip it per group in the edit pane (Full Width: off / show) — developers don't intervene.

className={`${theme.container} ${theme.layouts[group.full_width === 'show' ? 'fullwidth' : 'centered']}`}
group.full_width Class applied Result
'off' (default)layouts.centered · max-w-[1240px] mx-autoSections capped at the brand width inside the band
'show'layouts.fullwidth · '' (empty string)Sections fill the parent wrapper — no inner cap

The flag alone is not full-bleed. Setting full_width: 'show' on a content group still leaves the section inside wrapper2's own max-w-[1240px] cap. A true edge-to-edge canvas needs the workbench variant (wrapper2 w-full px-0) and full_width: 'show' so both caps drop. The portfolio map on properties.html is the surface that would use it: it renders at content width on the public page, and its staff counterpart on an app layout is the map workbench that goes full-bleed. Don't reach for workbench for ordinary content — the capped bands are correct for almost everything.
