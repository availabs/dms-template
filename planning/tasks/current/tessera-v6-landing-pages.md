# Set up the tessera.so landing, features, and docs pages on design_system_v6

## Objective

Build tessera.so's landing page, features page, and docs page as real DMS pages, transcribed from the finished `design_system_v6` ("Graph Paper") mockups, in the site's main pattern — and point that pattern's theme at the v6 theme.

## Before starting: read the skills folder

**Read `src/dms/skills/README.md` in full before writing anything** — it's the index over every DMS how-to, and this task touches theming, icons, page-building, and Card layout, so skimming only the list below risks missing a relevant recipe that was added since this task was written. The README's own "reading order for a new contributor" doubles as this task's dependency order: design-system grammar → theme translation → Card layout → page-building. The specific skills this task leans on hardest are listed under "Skills to read, in order" below — treat that as a floor, not a ceiling.

## Scope

**In scope:**
- 3 live DMS pages (draft): home/landing (`/`), features, docs — transcribed from `design_system_v6/pages/{beta-landing,features,docs}.html`.
- Setting the target pattern's `data.theme.selectedTheme` to the v6 theme.

**Out of scope** (per the v6 README's own scope note, and the draft-only discipline of the page-building skill):
- The editor/product UI kit, auth pages, additional example pages, production MapLibre/chart theming.
- Publishing — pages are created as **drafts only**; a human runs `dms page publish`.

Two consequences compound into the bug:
  1. `patterns/page/pages/edit/index.jsx:78-91` — on mount, if the resolved page has no `url_slug`, it treats that as "unconfigured" and client-side-redirects to `dataItems.sort_by_index().find(d => !d.parent && d.url_slug)` — the first *other* top-level page that has a real slug. Since `features` was created before `docs`, it won the redirect.
  2. Even setting `url_slug` to `""` wouldn't have fixed it: the path-builder in `_utils/index.js:109` does `d.url_slug || d.path || d.id`, and empty string is falsy, so it'd fall through to the numeric id (`/4`) instead of `/`.
  **Fix:** `dms page update 4 --set url_slug=/` — the literal string `"/"` is truthy (satisfies the edit-mode check) and the path-builder's `url.startsWith('/')` branch avoids a double slash, correctly producing `/` and `/edit/`. Verified live: `/edit` now shows "Page 1" selected in the sidebar and the real landing content. **This wasn't caught in the original verification pass** because I only ever navigated with `/edit` and it happened to still resolve correctly at first pass (before I'd created page 42/61, there was nothing else for the buggy fallback to redirect to) — the bug only manifested once sibling pages with real slugs existed. Worth remembering for future pattern/page scaffolding: **a home page must have `url_slug` explicitly set to `"/"`**, not left unset, or the pattern's own placeholder page needs this fixed at creation time.
